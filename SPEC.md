# event-sorcery 0.4.0 system specification

System specification for `event-sorcery`, version 0.4.0. This document replaces
the previous SPEC.md in full. Where `docs/domain.md`, `docs/cqrs.md`, or the
ADRs conflict with this document, this document wins; where they elaborate
consistently, they remain useful background. The concepts of ADR-0006
(cqrs-native durable jobs), ADR-0007 (reactor-side enqueue), ADR-0008
(entity-scoped durable operations), and ADR-0009 (handlers return events or one
job dispatch) are preserved and restated normatively here.

The key words MUST, MUST NOT, SHOULD, SHOULD NOT, and MAY are to be interpreted
as described in RFC 2119.

## 0. Status and scope

`event-sorcery` is **one engine with multiple language bindings**. The engine is
implemented once, in Rust, continuing this project's founding position: build
better abstractions on top of proven machinery (`cqrs-es`, `apalis`, SQLite)
rather than reimplementing it -- and, now, rather than reimplementing
_event-sorcery itself_ per language. Every storage backend, every core feature,
and every performance optimization is written once, in the engine. Bindings are
typed veneers: they own domain types, folds, and orchestration, and reach the
engine through a C-ABI FFI whose payloads are deterministic CBOR.

Two bindings are in scope:

- **Rust** -- the native binding: the typed API (`EventSourced`, `Store`,
  `Effect`, `DispatchedJob`, workers) living in the same workspace, lowering
  onto the engine's erased facade.
- **Haskell** -- a foreign binding over the FFI, implementing the same typed
  surface idiomatically (type families, linear one-shot values, sealed modules).

Normative artifacts:

- this document (`SPEC.md`);
- the DDL under `ddl/` (section 6);
- the generated C header `include/event_sorcery.h` (section 7; generated by
  cbindgen, but its declared surface is contract);
- the conformance vectors under `conformance/` (section 10).

Everything else -- crate/package layout, module names, docs -- is per-component
and informative.

## 1. Design position

### 1.1 Goals

- **One engine.** Storage, transactions, fencing, job semantics, checkpoints,
  schema reconciliation: implemented once. New backends (Postgres) and
  optimizations land once and every binding inherits them.
- **Capture invariants in types -- in each binding's own type system.**
  Aggregate identity, lifecycle, job membership, sealed verdicts, one-shot
  protocol values.
- **Fallible apply.** `originate`/`evolve` return errors; failure is part of the
  persisted lifecycle, never a panic.
- **Schema drift detection.** Version bumps invalidate stale snapshots and views
  on startup, with an explicit refusal where invalidation would destroy
  compacted state.
- **Durable side effects.** Handlers are pure; external actions run as durable,
  fenced, at-least-once jobs; verdicts are sealed proofs delivered back into
  entity state.
- **A boring boundary.** No transaction ever spans the FFI; the engine never
  calls back into a binding; every call is one atomic load-decide-enact.

### 1.2 Non-goals

- A distributed event store. A store is a single database owned by one service;
  services integrate via section 9. Within a service, multiple processes
  (including different-language processes) may share a store under section 6.7.
- A general-purpose CQRS framework, broker, or workflow engine.
- Multi-tenancy at the framework level.
- Online migration of the events table. Event shapes are permanent.
- Engine callbacks. The ABI is strictly call-in; anything requiring the engine
  to invoke binding code (inversion of control) is out of scope by design.
- Cross-language interpretation of _domain_ payloads. Entity events and job
  payloads are opaque bytes owned by the service that defines them; the spec
  makes their wrappers portable, not their contents.

## 2. Architecture

```text
      Haskell binding                        Rust binding (native)
EventSourced / Store / Effect          EventSourced / Store / Effect
DispatchedJob (sealed) / workers       DispatchedJob (sealed) / workers
reactor & projection folds             reactor & projection folds
      |            (typed: domain folds, orchestration)        |
      |  CBOR over C ABI                          direct calls |
      v                                                        v
+--------------------- erased engine facade ---------------------+
|   streams  jobs  outbox  checkpoints  projections  schema      |
|   receipts snapshots  -- all state transitions, all storage    |
|   (kernels section 5, storage section 6; cqrs-es/apalis/SQLite) |
+------------------------------------------------------------------+
                  event-sorcery-ffi (staticlib + header)
```

- **The seam** is the storage/decision boundary: everything below it
  (persistence, transactions, kernel decisions, fencing, polling state) is
  engine; everything above it (entity folds, command routing, effect
  construction, job `submit`/`reconcile` bodies, reactor reactions, projection
  `apply`, delivery to brokers) is binding.
- **Control flow is one-directional.** Bindings call the engine; the engine
  NEVER calls a binding. The ABI contains no callbacks, no function pointers.
  Consequences: no foreign-runtime reentrancy, no cross-runtime deadlocks, and
  the engine is testable as a black box.
- **The erased facade is the only writer.** The Rust typed binding MUST lower
  through the same facade the FFI exports; it gets no private side door. This
  makes identical binding semantics a structural fact rather than a testing
  goal.
- **No transaction spans the FFI.** Each engine call that writes is one atomic
  transaction (section 6.3). Bindings compose calls through the same
  optimistic/fenced protocols any external process would use; there is no
  distributed state across the boundary.

## 3. Model

### 3.1 Streams, events, offsets

- A **stream** is identified by
  `StreamIdentity = (aggregate_type: text,
  aggregate_id: text)`.
- Events within a stream carry a **sequence**: a `uint` starting at **1**,
  strictly contiguous. Replay (section 8.1) MUST reject gaps. No legacy gap
  tolerance.
- Every event receives a store-wide **global offset**: strictly increasing in
  commit order across all streams, **monotonic but not necessarily dense**. No
  component may assume density.
- An event row is
  `(global_offset, aggregate_type, aggregate_id, sequence,
  event_type: text, event_version: uint, payload: bytes)`.
- **Domain payloads are opaque to the engine.** The engine stores and returns
  bytes; metadata validation and decoding happen in the binding (section 8.1).
- Events are immutable; never updated or deleted, except compaction (section
  6.8) for entities that opted into `CompactAfterSnapshot`.

### 3.2 Time

All instants are `uint` milliseconds since the Unix epoch. Engine kernels never
read a clock: time-dependent operations take instants as call arguments
(`now_ms`, lease windows, `run_at`). Bindings inject clocks and MUST make them
injectable for tests.

`LeaseWindow = (start, end)` with `start < end` enforced at construction;
`start` is the claimant's now, `end` the proposed lease expiry.

### 3.3 Framework streams

`aggregate_type`s beginning with `$` are reserved for the engine. Bindings MUST
reject entity definitions with `$`-prefixed `AGGREGATE_TYPE`. Defined:

| Stream | Identity           | Purpose                                                |
| ------ | ------------------ | ------------------------------------------------------ |
| `$job` | `("$job", job_id)` | Audit trail of job lifecycle transitions (section 3.6) |

Framework streams are excluded from transport relays (section 9.3) and skipped
by domain folds via `aggregate_type` filters. The journal reader
(`es_journal_read`) returns them; consumers filter.

### 3.4 Entities (binding-side)

An **entity** is a binding-side domain type implementing that binding's
`EventSourced` surface, declaring: a strongly typed `Id` (text-convertible); an
`Event` type with per-event `event_type: text` and `event_version: uint`;
`Command` and error types; a `Jobs` list of dispatchable job types (dispatching
an undeclared job type MUST be a compile-time error in bindings whose type
system can express it -- both in-scope bindings can); a stable `AGGREGATE_TYPE`
(never `$`-prefixed); a `SCHEMA_VERSION: uint`; a compaction policy (`Retain`
default, `CompactAfterSnapshot`); pure event-side functions `originate`/`evolve`
(fallible); pure command-side functions `initialize`/`transition` returning
`Effect`.

The **lifecycle** of a stream is `Uninitialized -> Live(entity)`, with `Failed`
absorbing any `originate`/`evolve` failure or metadata/sequence violation during
replay. A `Failed` lifecycle rejects all further commands with its stored error.
The lifecycle fold runs entirely in the binding; the engine sees only bytes and
versions.

### 3.5 Effects

`Effect entity = Events (NonEmpty (Event entity)) | Dispatch job`, `job` a
member of the entity's `Jobs` list. Exactly one dispatch, or events -- never
both, never more than one dispatch. Handlers perform no I/O. Interpretation and
commit are specified as the command protocol, section 8.2.

### 3.6 Jobs (engine-side records, binding-side execution)

A **job** is durable at-least-once background work. Its operational truth is an
engine-held `JobRecord`:

```text
JobRecord = { payload  : bytes
            , status   : Ready | Scheduled(run_at) | Leased(expires)
                       | Completed | DeadLettered(reason)
            , token    : uint
            , attempts : uint
            , claims   : uint
            }
DeadReason = RetriesExhausted | Rejected | Undecodable | Abandoned
```

Fixed semantics (kernels section 5.3):

- **Fencing is the monotonic token.** Every successful claim increments `token`;
  every post-execution transition presents the claiming token and is refused
  (`LeaseLost`) if superseded. `token = 1` after a claim means _first execution
  ever_ -- the basis of submit/reconcile routing.
- **A crash is not a failed attempt.** `attempts` advances only on recorded
  failures.
- **The claim budget catches crash loops.** `claims` increments per claim;
  exceeding the budget dead-letters `Abandoned` at claim time. **Defer resets
  `claims` to 0** (a defer is a productive checkpoint; pollers may defer
  indefinitely).
- **Terminal states are retained.**
- **Audit trail.** Every record-changing job transition appends one lifecycle
  event to the job's `$job` stream in the same transaction: `event_type` in
  {`enqueued`, `claimed`, `succeeded`, `retry-scheduled`, `deferred`,
  `dead-lettered`}, `event_version = 1`, payload = post-transition `JobRecord`
  (section 4 CBOR). Idempotent no-op transitions append nothing.

Two job kinds share the machinery:

- **Standalone jobs** -- origin-less `perform`; enqueued directly (own
  transaction) by reactors, pollers, chains, startup recovery; optional delayed
  start (`run_at`).
- **Entity-dispatched jobs** -- kicked off only via `Effect::Dispatch`; carry an
  `Origin` entity; execute via `submit`/`reconcile`; settle a verdict onto the
  origin (sections 3.7 and 8.4).

Job payloads use the `stored-job` wrapper (section 4.2) around an opaque
job-defined payload. Failure classification is explicit at every return site
(`Transient` retries with backoff; `Terminal` dead-letters `Rejected`); bindings
MUST NOT provide implicit conversions that let monadic bind/`?` classify
silently.

Execution bodies (`perform`/`submit`/`reconcile`) run in the binding that
defines the job type. Job **kinds are partitioned by configuration**: each
binding's workers poll only the kinds they define. Concurrent executors for one
kind are fenced and legal; same-binding executors are expected.

### 3.7 Entity-dispatched jobs

Embedded in origin-entity state, one machine per dispatched-job field:

```text
DispatchedState  = Idle | InFlight(job_id)
                 | Confirmed(Settled)
                 | Failed(SettledFailure)
Settled          = (job_id, output, attempts)
SettledFailure   = (job_id, failure, attempts)
DispatchFailure  = Rejected(domain_error) | DeadLettered(reason, detail: text)
DispatchEvent    = Dispatched(job_id, job) | ConfirmedEv(Settled)
                 | FailedEv(SettledFailure)
```

- **Guard**: fresh dispatch legal from `Idle` or `Failed`; refused `InFlight`
  (overlap) or `Confirmed` (already done).
- **The intent is the job**: `Dispatched` carries the job value and minted
  `job_id`; the binding's command protocol (not the handler) emits it and
  includes the job seed in the same commit (section 8.2), so intent and enqueue
  are atomic and no free-form event can accompany an enqueue.
- **Verdicts are sealed**: `Settled`, `SettledFailure`, and the
  `DispatchOutcome` command wrapper are constructible only by the binding's
  verdict-delivery path, enforced by visibility; test fabrication gated behind
  an explicit test-only feature/module. `Confirmed` in entity state is a proof,
  not a claim.
- **Settle** correlates on `job_id`, absorbs duplicate delivery of an
  already-settled verdict (no events, no error), refuses everything else
  (`OutcomeMismatch`).
- A dispatched job type MUST expose the origin's identity from its own payload
  (verdict delivery does no side lookups); the origin's event and command types
  absorb `DispatchEvent`/`DispatchOutcome` via the binding's injection
  mechanism.
- **Submit/reconcile routing**: first execution runs `submit`; later executions
  run `reconcile` first -- `Settled(o)` (earlier call landed), `NotSubmitted`
  (provably never reached the external system; authorizes `submit` in the same
  execution), `Indeterminate(delay)` (defer; no attempt counted). Routing
  over-approximates safely: never `submit` while a prior submission might exist.
  External idempotency keys derive from `job_id`, never the attempt number.

The full state tables are K12-K14 (section 5.5); although this machine executes
in bindings, its transitions are normative and identical across bindings.

### 3.8 Reactors and the outbox

A **reactor** is a named, checkpointed, **pull-based** consumer of the global
journal: the binding folds `es_journal_read(checkpoint, ...)` in offset order,
runs its reaction on events of its declared entity types, and advances the
engine-held checkpoint per event. A reaction failure stops the run at that
offset (checkpoint does not advance past it); rerunning resumes there.
Synchronous post-commit notification MAY exist as a latency optimization for
projections but MUST NOT be the durability mechanism: the checkpoint is.

A reactor's only durable output is at most one **outbox entry** per event:

```text
OutboxEntry   = (delivery_id: bytes32,
                 payload: CommandDelivery(bytes) | JobDispatch(bytes))
OutboxRecord  = { entry, status, token, attempts, claims }
OutboxStatus  = Ready | Scheduled(run_at) | Leased(expires)
              | Delivered | DeadLettered(OutboxDeadReason)
OutboxDeadReason = RetriesExhausted | Rejected | Abandoned
```

Checkpoint advance and outbox insertion are one atomic engine decision (K15):
identical re-proposal absorbs (crash between insert and checkpoint is safe);
divergent re-proposal for one `delivery_id` is a hard error. Outbox delivery
kernels mirror the job kernels (claim budget included; no defer; no
`Undecodable` -- an undecodable framework wrapper is `Rejected`). Delivery
targets absorb duplicates via receipts, so the outbox is at-least-once with
exactly-once effect downstream.

### 3.9 Projections

A **projection** is a named, checkpointed fold of the journal into a view:
binding-side `(name, schema_version, initial, apply, encode, decode)`,
engine-side state `(event_offset, view_bytes)`. Checkpoints are **monotonic, not
dense** (K22). Skip-protection is the reader's contract: the journal reader
yields every event with `global_offset > cursor` ascending. Bindings validate
envelope metadata before applying, and rebuild by reset plus catch-up.

### 3.10 Schema registry

An engine-held table `(kind: 'aggregate' | 'projection', name,
schema_version)`
-- not an aggregate. Startup reconciliation (K23 and section 6.9): unregistered
-> register, preserve; equal -> nothing; changed -> invalidate derived state
(aggregate: clear snapshots -- **refused with an explicit error if the aggregate
compacts**; projection: reset), recover, and only after recovery durably
completes, record the version. A crash before recording re-runs the idempotent
recovery.

### 3.11 Delivery receipts

Engine-held `delivery_receipts(delivery_id)`. Recorded -> `AlreadyApplied`
(apply nothing); else apply-and-record in one transaction (K24 and
`es_commit_delivery`). The universal exactly-once-effect primitive: command
delivery, verdict-free integrations, and bus ingestion terminate here.

### 3.12 Responsibility split

| Concern                                                         | Engine | Binding |
| --------------------------------------------------------------- | ------ | ------- |
| Event storage, sequences, global offsets, commit atomicity      | yes    |         |
| Concurrency check (K1), commit limits                           | yes    |         |
| Job/outbox records, transitions, fencing, budgets, audit events | yes    |         |
| Checkpoints, projections, schema, receipts, snapshots           | yes    |         |
| Poll queries (runnable jobs/outbox)                             | yes    |         |
| Entity types, lifecycle fold, replay validation                 |        | yes     |
| Command routing, effect interpretation, batch construction      |        | yes     |
| Job bodies and worker loop orchestration                        |        | yes     |
| Dispatch machine state, sealed verdicts, verdict delivery       |        | yes     |
| Reactor reactions, projection apply, view codecs                |        | yes     |
| Broker clients, relay/ingest loops                              |        | yes     |
| Clocks, backoff schedules, retry limits                         |        | yes     |

## 4. Encoding

### 4.1 Deterministic CBOR profile

Every ABI payload and every engine-held framework structure encodes as RFC 8949
section 4.2.1 deterministic CBOR, further restricted:

- definite-length items only; integers in shortest form; **no floats** in
  framework types; no tags;
- sum types encode as arrays `[tag: uint, ...args]` -- never string tags;
- maps only where the CDDL below says so, keys sorted bytewise;
- every top-level framework type carries a leading **format version** integer,
  independent of domain schema versions.

Bindings SHOULD implement codecs by hand (explicit list-length encode/decode,
auditable line-by-line against the CDDL) rather than by derivation.

conflicts).

### 4.3 Identity derivations

`delivery_id` for any delivery derived from a journal position:

```text
delivery_id = SHA-256( lp(source_service) || lp(aggregate_type)
                     || lp(aggregate_id)   || be64(sequence) )
where lp(s) = be64(len(utf8(s))) || utf8(s)
```

It is deterministic, derivable by producer and every consumer, and stable across
redelivery and replays. `source_service` is the store-owning service's stable
configured name.

`JobId` is opaque non-empty UTF-8 text at most 128 bytes, unique per dispatch or
enqueue; ULID RECOMMENDED. Job streams are `("$job", job_id)`. Hand-constructed
deliveries MAY use any 32-byte identifier; journal-derived deliveries MUST use
the derivation above.

## 5. Engine semantics (kernels)

The engine's behavior is specified as pure decision functions over records. They
are implemented once, inside the engine, as a dedicated pure `kernel` module,
with storage code reduced to load-decide-enact around them. The engine's unit
tests replay the kernel vectors of section 10.2 against them directly. **Check
order is normative**: where a state matches several conditions, the outcome is
the FIRST matching line. Error identity is part of the ABI; bindings
pattern-match on these encodings.

Shared guard for K5-K9:

```text
LEASED-GUARD(token, record?)
  1  record absent                    -> err NotFound
  2  token != record.token            -> err LeaseLost(token, record.token)
  3  status Ready|Scheduled(_)        -> err NotLeased
  4  status Completed                 -> err AlreadyCompleted
  5  status DeadLettered(reason)      -> err AlreadyDeadLettered(reason)
  6  status Leased(_)                 -> ok
```

### 5.1 Catalog

| Number  | Kernel                                         | Engine operations                              |
| ------- | ---------------------------------------------- | ---------------------------------------------- |
| K1      | append check                                   | `es_commit`, `es_commit_delivery`              |
| K3      | job enqueue                                    | `es_job_enqueue`, job-seed slot of `es_commit` |
| K4      | job claim                                      | `es_job_claim`                                 |
| K5-K9   | ack, retry, defer, dead-letter, exhaust        | `es_job_ack` through `es_job_exhaust`          |

| K16-K20 | outbox claim, ack, retry, dead-letter, exhaust | `es_outbox_*`                                  |
| K22     | projection advance                             | `es_projection_advance`                        |
| K23     | schema reconcile                               | `es_schema_reconcile`                          |
| K24     | receipt check                                  | `es_commit_delivery`, `es_delivery_recorded`   |
| SR      | snapshot resolve                               | `es_snapshot_resolve`                          |

K2 replay validation, K10 routing, K11 disposition, K12-K14 dispatch, and K21

executed binding-side (section 8).

### 5.2 Stream kernels

```text
K1 APPEND_CHECK(expected, actual)
  expected == actual -> ok
  otherwise          -> err ConcurrencyConflict(stream, expected, actual)
```

Applied per stream append in a commit batch; all-or-nothing with the batch. The
storage arbiter is the events UNIQUE constraint (section 6.1). Commit limits
(section 6.4) are checked before any write.

### 5.3 Job kernels

```text
K3 ENQUEUE(payload, run_at?, record?)
  1  absent, run_at absent  -> (Enqueued, Ready, token 0, attempts 0, claims 0)
  2  absent, run_at present -> (Enqueued, Scheduled(run_at), token 0,
                                attempts 0, claims 0)
  3  payload == record.payload -> (AlreadyEnqueued, record)
  4  otherwise                 -> err PayloadMismatch

K4 CLAIM(window, budget, record?)
  1  absent                                  -> err NotFound
  2  status Completed                        -> err AlreadyCompleted
  3  status DeadLettered(reason)             -> err AlreadyDeadLettered(reason)
  4  Scheduled(time), window.start < time    -> err NotRunnable(time)
  5  Leased(time), window.start < time       -> err LeaseUnavailable(time)
  6  claims == 2^64 - 1                      -> err ClaimCountExhausted
  7  claims + 1 > budget                     -> (Abandon,
                                                  DeadLettered(Abandoned),
                                                  claims + 1)
  8  token == 2^64 - 1                       -> err LeaseTokenExhausted
  9  otherwise                               -> (Claimed(token + 1, attempts,
                                                   payload),
                                                  Leased(window.end),
                                                  token + 1, claims + 1)

K5 ACKNOWLEDGE(token, record?)
  1  absent                     -> err NotFound
  2  token != record.token      -> err LeaseLost(token, record.token)
  3  Ready|Scheduled(_)         -> err NotLeased
  4  Leased(_)                  -> status = Completed
  5  Completed                  -> record
  6  DeadLettered(reason)       -> err AlreadyDeadLettered(reason)

K6 RETRY(token, run_at, record?)
  LEASED-GUARD
  attempts' = attempts + 1 (overflow -> err AttemptCountExhausted)
  -> (attempts', status = Scheduled(run_at), attempts = attempts')

K7 DEFER(token, run_at, record?)
  LEASED-GUARD -> status = Scheduled(run_at), claims = 0

K8 DEAD_LETTER(token, reason, record?)
  0  already DeadLettered(reason), token matches -> record
  otherwise LEASED-GUARD -> status = DeadLettered(reason)

K9 EXHAUST(token, record?)
  LEASED-GUARD
  attempts' = attempts + 1 (overflow -> err AttemptCountExhausted)
  -> (attempts', status = DeadLettered(RetriesExhausted),
      attempts = attempts')
```

Invariants: `token` strictly increases across successful claims; claims never
modify `attempts`; only K7 resets `claims`; renewal (`es_job_renew`) extends
`Leased(expires)` without touching `token` and reports `Lost` when the token was
superseded.

### 5.4 Checkpoint, schema, receipt, and snapshot kernels

```text
K15 REACTOR_ADVANCE(requested, proposed?, checkpoint?, existing?)
  checkpoint_value = checkpoint or 0
  1  requested <= checkpoint_value -> (AlreadyCommitted; no writes)
  2  proposed absent               -> (Committed, checkpoint = requested)
  3  existing absent               -> (Committed, checkpoint = requested,
                                        insert ready outbox record)
  4  existing == proposed          -> (Committed, checkpoint = requested)
  5  existing != proposed          -> err DeliveryMismatch(delivery_id)

K16-K20 are textually K4, K5, K6, K8, and K9 with JobRecord replaced by
OutboxRecord, payload replaced by entry, and DeadReason replaced by
OutboxDeadReason. There is no defer and no Undecodable reason.

K22 PROJECTION_ADVANCE(requested, view, state?)
  checkpoint_value = state.offset or 0
  1  requested <= checkpoint_value -> (AlreadyApplied, state)
  2  otherwise                     -> (Advanced, {requested, view})

K23 SCHEMA_RECONCILE(requested, current?)
  current absent       -> (Registered, PreserveDerivedState)
  current == requested -> (Current, PreserveDerivedState)
  otherwise            -> (Changed(current), InvalidateDerivedState)

K24 RECEIPT_CHECK(recorded)
  true  -> AlreadyApplied
  false -> Apply

SR SNAPSHOT_RESOLVE(current_version, snapshot?)
  1  snapshot absent                              -> FullReplay
  2  schema_version == current_version             -> Resume(sequence, payload)
  3  mismatch, history = retained                  -> discard; FullReplay
  4  mismatch, history = compacted                 -> err SnapshotSchemaMismatch
```

Checkpoints are monotonic, not dense. `requested == checkpoint + 1` checks are
non-conforming. Skip-protection is the journal reader's contract.

### 5.5 Binding-executed kernels

Executed identically in both bindings:

```text
K2 REPLAY_STEP(entity?, expected_position, stored_event)
  1  sequence != expected_position -> err SequenceMismatch(expected, actual)
  2  declared event metadata mismatch -> err EventTypeMismatch or
                                         EventVersionMismatch
  3  decode(payload) fails -> err DecodeFailed(position, cause)
  4  absent -> originate(event); present -> evolve(entity, event)
     Left error -> err ApplicationFailed(position, error)
  5  -> (entity', expected_position + 1)

K10 ROUTE(token)
  token == 1 -> Submit
  token >= 2 -> Reconcile

K11 DISPOSITION(limit, attempts)
  next = attempts + 1 (overflow -> err)
  next >= limit -> Exhaust(next)
  otherwise     -> Retry(next)

K12 DISPATCH_GUARD(state)
  Idle | Failed(_) -> ok
  InFlight(_)      -> err InFlight
  Confirmed(_)     -> err AlreadyConfirmed

K13 SETTLE(verdict, state)
  1  InFlight(job), Confirm(settled), ids equal -> [ConfirmedEv(settled)]
  2  InFlight(job), Fail(failure), ids equal    -> [FailedEv(failure)]
  3  Confirmed(settled), matching Confirm       -> []
  4  Failed(failure), matching Fail             -> []
  5  otherwise                                  -> err OutcomeMismatch

K14 DISPATCH_FOLD(state, event)
  1  (Idle | Failed(_)) x Dispatched(job, _)       -> InFlight(job)
  2  InFlight(job) x matching ConfirmedEv(settled) -> Confirmed(settled)
  3  InFlight(job) x matching FailedEv(failure)    -> Failed(failure)
  4  otherwise                                     -> err DispatchReplay

K21 CLASSIFY_ENQUEUE(result)
  Right (Enqueued | AlreadyEnqueued) -> ok
  Left engine-error class STORAGE    -> TransientDelivery(error)
  Left otherwise                     -> TerminalDelivery(error)
```

K10 is sound because `token = 1` if and only if no claim ever succeeded before,
and only claimed executions can reach an external system. A crash after claim,
before the external call, yields `token >= 2` on reclaim and therefore
`reconcile`.

## 6. Storage

### 6.1 Logical schema

Normative DDL lives in `ddl/sqlite/` and `ddl/postgres/`. The SQLite dialect is:

```sql
CREATE TABLE events (
  global_offset  INTEGER PRIMARY KEY AUTOINCREMENT,
  aggregate_type TEXT    NOT NULL,
  aggregate_id   TEXT    NOT NULL,
  sequence       INTEGER NOT NULL,
  event_type     TEXT    NOT NULL,
  event_version  INTEGER NOT NULL,
  payload        BLOB    NOT NULL,
  UNIQUE (aggregate_type, aggregate_id, sequence)
) STRICT;

CREATE TABLE jobs (
  job_id        TEXT PRIMARY KEY,
  record        BLOB NOT NULL,
  kind          TEXT NOT NULL,
  status        TEXT NOT NULL,
  run_at        INTEGER,
  lease_expires INTEGER
) STRICT;
CREATE INDEX jobs_poll ON jobs (kind, status, run_at, lease_expires);

CREATE TABLE outbox (
  delivery_id   BLOB PRIMARY KEY,
  record        BLOB NOT NULL,
  status        TEXT NOT NULL,
  run_at        INTEGER,
  lease_expires INTEGER
) STRICT;
CREATE INDEX outbox_poll ON outbox (status, run_at, lease_expires);

CREATE TABLE delivery_receipts (delivery_id BLOB PRIMARY KEY) STRICT;
CREATE TABLE reactors (name TEXT PRIMARY KEY, event_offset INTEGER NOT NULL) STRICT;

CREATE TABLE projections (
  name TEXT PRIMARY KEY,
  event_offset INTEGER NOT NULL,
  view BLOB NOT NULL
) STRICT;

CREATE TABLE snapshots (
  aggregate_type TEXT    NOT NULL,
  aggregate_id   TEXT    NOT NULL,
  sequence       INTEGER NOT NULL,
  schema_version INTEGER NOT NULL,
  history        TEXT    NOT NULL CHECK (history IN ('retained', 'compacted')),
  payload        BLOB    NOT NULL,
  PRIMARY KEY (aggregate_type, aggregate_id)
) STRICT;

CREATE TABLE schemas (
  kind           TEXT    NOT NULL CHECK (kind IN ('aggregate', 'projection')),
  name           TEXT    NOT NULL,
  schema_version INTEGER NOT NULL,
  PRIMARY KEY (kind, name)
) STRICT;
```

### 6.2 Record-authoritative rule

For `jobs` and `outbox`, the CBOR `record` column is the truth the kernels read;
`kind`, `status`, `run_at`, and `lease_expires` are a denormalized poll index
written **in the same transaction, derived from the record**. Pollers read the
index; kernels read the record; they cannot disagree. Lease renewal updates
`record` and `lease_expires` together, keyed on the current token.

### 6.3 Transactional boundaries

Every write operation is one engine transaction holding load and enact together
(SQLite uses `BEGIN IMMEDIATE` or stronger, so the load reads under the write
lock):

| Engine operation         | Reads                    | Writes in one transaction                                   |
| ------------------------ | ------------------------ | ----------------------------------------------------------- |
| `es_commit`              | stream versions          | event rows; optional job row and `$job` enqueue audit event |
| `es_commit_delivery`     | receipt, stream versions | events, optional seed, receipt                              |
| job kernel operations    | job row                  | job row and `$job` audit event unless unchanged             |
| `es_job_enqueue`         | job row                  | job row and enqueue audit event                             |
| `es_reactor_advance`     | checkpoint, outbox row   | checkpoint and optional outbox insert                       |
| outbox kernel operations | outbox row               | outbox row                                                  |
| `es_projection_advance`  | projection row           | projection row                                              |
| `es_snapshot_store`      | stream version, snapshot | snapshot upsert                                             |
| schema operations        | schema row               | per section 6.9                                             |

Poll queries are plain reads:
`status = 'ready' OR (status = 'scheduled' AND run_at <= now) OR
(status = 'leased' AND lease_expires < now)`.
A stale poll hit degrades to a claim-time refusal, never a double claim.

### 6.4 Commit limits

Configured at `es_open` (`payload_limit`, `batch_limit`, both positive).
`es_commit` and `es_commit_delivery` MUST reject, before any write:
`EventPayloadTooLarge`, `BatchEventLimitExceeded`, and `DuplicateStreamInBatch`.

### 6.5 SQLite profile

`journal_mode = WAL`, `busy_timeout >= 5000` ms, `synchronous = FULL`, and
`STRICT` tables. `":memory:"` is a legal test path. The engine serializes its
own writes internally; correctness rests on `BEGIN IMMEDIATE` plus kernels, not
the serialization.

### 6.6 Postgres dialect

The Postgres dialect has the same logical schema:
`BIGINT GENERATED ALWAYS AS
IDENTITY` for `global_offset`, `BYTEA` for `BLOB`,
identical constraints, and millisecond instants stored as `BIGINT`. Write
transactions use row locks or serializable isolation. The events UNIQUE
constraint remains the append arbiter. A Postgres backend is an engine feature;
bindings inherit it through open options without changes.

### 6.7 Cohabitation

Every write path is kernel-fenced (events UNIQUE, token comparison, monotonic
checkpoints, receipt primary keys), so no interleaving of conforming writers
violates the invariants. Unlimited read-only attachments to a live store are
legal and RECOMMENDED. Multiple writing processes, including different-language
processes through their own engine instances, are legal; one writing process per
store is RECOMMENDED operationally. Between services, stores are private and
integration uses section 9.

### 6.8 Snapshots and compaction

`StoredSnapshot = (sequence, schema_version, history, payload)`. Write
validation requires `sequence <=` current stream version and never regresses an
existing snapshot. Snapshot resolution is engine-enacted (SR): a mismatched
compacted snapshot is a hard error; a mismatched retained snapshot is discarded
and full replay proceeds.

Compaction (`CompactAfterSnapshot` entities only; `es_compact`) MAY delete
events with `sequence <= snapshot.sequence`. Entities with projections MUST be
`Retain`.

### 6.9 Startup reconciliation

For each registered target, in order:

1. `es_schema_reconcile(kind, name, requested, compacts)` runs K23 and enacts
   invalidation. A changed aggregate that compacts and has a stored version is
   refused with `CompactedSnapshotClear`; otherwise its snapshots are cleared. A
   changed projection is reset.
2. The binding recovers: changed projections rebuild from zero; current
   projections catch up only.
3. Only after recovery durably completes, `es_schema_record` stores the version.

A crash before step 3 re-runs the idempotent sequence.

## 7. ABI

The engine ships as crate `event-sorcery-ffi` with `crate-type = ["staticlib"]`;
a `cdylib` MAY also be offered. A cbindgen-generated header at
`include/event_sorcery.h` is produced as a build artifact and is never checked
in or hand-edited. SQLite is bundled so the archive is self-contained apart from
platform libraries.

### 7.1 Handles and buffers

```c
typedef struct EsStore EsStore;
typedef struct { uint8_t *ptr; size_t len; } EsBuf;
```

- Input buffers are caller-owned and valid only for the call; the engine copies
  anything it retains.
- Output buffers are engine-allocated and the caller MUST free each exactly once
  via `es_buf_free`. A zero-length output has `ptr = NULL`; freeing it is a safe
  no-op.
- `EsStore` is thread-safe. Calls on one handle from any number of threads are
  legal; the engine serializes internally per section 6.3.

### 7.2 Threading and runtime

- Every call is synchronous and may block. The engine owns a captive
  multi-thread tokio runtime per store, sized by `runtime_threads`, and the
  calling thread parks until completion.
- The engine NEVER calls foreign code. There are no callbacks, function
  pointers, or logging callbacks. Logging uses an engine-owned `tracing`
  subscriber configured by `log_filter`.
- The engine installs no signal handlers.

### 7.3 Errors and panics

Every export returns `int32_t`: zero means success; otherwise it is an error
class, with the output buffer containing an `engine-error`:

| Class | Name              | Detail                                   |
| ----- | ----------------- | ---------------------------------------- |
| 1     | `ES_ERR_DECODE`   | text cause for malformed request payload |
| 2     | `ES_ERR_CONFLICT` | `commit-conflict`                        |
| 3     | `ES_ERR_JOB`      | `job-error`                              |
| 4     | `ES_ERR_OUTBOX`   | `outbox-error`                           |
| 5     | `ES_ERR_REACTOR`  | `reactor-error`                          |
| 6     | `ES_ERR_SCHEMA`   | compacted-clear or typed schema detail   |
| 7     | `ES_ERR_SNAPSHOT` | invalid version or compacted mismatch    |
| 8     | `ES_ERR_LIMIT`    | `limit-violation`                        |
| 9     | `ES_ERR_STORAGE`  | transient backend error chain            |
| 10    | `ES_ERR_STATE`    | null, closed, or poisoned handle         |
| 100   | `ES_PANIC`        | empty                                    |

Every export wraps its body in `catch_unwind`; a panic unwinding across
`extern "C"` aborts the process. After `ES_PANIC`, the handle is poisoned and
all subsequent calls return `ES_ERR_STATE`; the binding closes and reopens.
`ES_ERR_STORAGE` is the transient class. All other classes are non-retryable at
the call site.

### 7.4 Versioning

```c
uint32_t es_abi_version(void);
```

The result is `(major << 16) | minor`. Bindings MUST check major equality in
their open wrapper and refuse to proceed on mismatch. Additive exports and
additive CBOR fields guarded by format-version bumps increment minor; anything
else increments major.

### 7.5 Lifecycle exports

```c
int32_t es_open (EsBuf options, EsStore **out);
int32_t es_close(EsStore *store);
void    es_buf_free(EsBuf buf);
```

Bindings hold the handle behind a finalizer-equipped owner (`ForeignPtr` or
`Drop`) calling `es_close`. Explicit close is legal and close is idempotent.

## 8. Binding protocols

These normative orchestrations execute identically in every binding and compose
only section 7 calls with section 5.5 binding-side kernels.

### 8.1 Replay and load

`loadEntity(id)` calls `es_snapshot_resolve(type, id, SCHEMA_VERSION)`.
`FullReplay` loads the full stream and folds K2 from position 1 on
`Uninitialized`. `Resume(sequence, payload)` decodes the snapshot (failure is a
hard error), loads events after `sequence`, and folds K2 from `sequence + 1`.
Any K2 error marks the lifecycle `Failed`.

### 8.2 Command protocol

`executeCommand(id, command)`:

1. Load per section 8.1 and retain the resulting expected version.
2. Route `Uninitialized` to `initialize`, `Live` to `transition`, and reject a
   `Failed` lifecycle with its stored error.
3. Interpret the `Effect`:
   - `Events events`: fold them through K2's application step from current
     state. An event rejected by its own fold fails the command. Build one
     append with no job seed.
   - `Dispatch job`: run K12 against its dispatch field; mint `job_id`; build
     and fold `Dispatched(job_id, job)` through the entity injection; build one
     append with job seed `[job_id, stored-job, null]`.
4. Call `es_commit`. A conflict is classified by stream identity as entity,
   `$job`, or unknown. Intent event and enqueue are atomic.

`deliverCommand(delivery_id, id, command)` is the same protocol ending in
`es_commit_delivery`. A receipt pre-read MAY be used as a fast path, but the
commit-time receipt check is authoritative.

### 8.3 Worker loop

For each configured job kind and
`(lease_ms, budget, max_attempts, backoff,
delivery_policy)`:

1. Poll `es_job_poll(kind, now, limit)`.
2. Claim each id with `es_job_claim(id, now, now + lease_ms, budget)`. Refusals
   are skipped. `Abandon` frees the slot and is reported.
3. Decode `stored-job` and the job payload. Failure dead-letters Undecodable.
4. K10 routes a dispatched job by token. `Submit` invokes `submit`; `Reconcile`
   invokes `reconcile`, then maps `Settled` to done, `NotSubmitted` to submit in
   the same execution, and `Indeterminate` to defer. Standalone jobs always
   invoke `perform`.
5. Persist: done -> acknowledge; defer -> defer; terminal -> dead-letter
   Rejected; transient -> K11 and retry or exhaust.
6. Long executions renew well before lease expiry. Renewal stops before step 5.
   On `Lost`, step 5 is skipped. A lost lease on successful work is logged as a
   double-execution alarm at error severity.

Backoff is exponential with a cap and uses saturating arithmetic.

### 8.4 Verdict delivery

This protocol interleaves with section 8.3 step 5 for entity-dispatched jobs:

- `Done(output)`: construct the sealed
  `Confirm(Settled(job_id, output, attempts + 1))` and deliver it as a
  `DispatchOutcome` command to the origin before acknowledging. A transport or
  storage failure defers on `retry_delay`; origin refusal defers on
  `refused_backoff` and logs at error severity. Neither counts an attempt.
- `Terminal(error)`: deliver a sealed rejected failure before dead-lettering.
  Failed delivery defers; a rejection cannot dead-letter undelivered.
- `Exhaust(attempts)`: best-effort deliver a retries-exhausted failure, then
  exhaust regardless. Failed delivery logs `DISPATCH DANGLING`.
- Claim-budget abandonment happens before execution and delivers nothing. It is
  the same dangling class.

### 8.5 Reactor and projection folds

Reactor catch-up loads its checkpoint and reads journal pages in offset order.
It skips framework streams and undeclared aggregate types while still advancing
the checkpoint. For declared types, it validates metadata, decodes, runs the
reaction to an optional outbox entry with a section 4.3 delivery id, and calls
`es_reactor_advance`. Reaction failure stops the run.

Outbox delivery polls, claims, delivers, and settles like the worker protocol.
`JobDispatch` calls `es_job_enqueue` and K21. `CommandDelivery` uses either
transport or local receipt-guarded command execution.

Projection catch-up mirrors the reactor fold with projection load, apply, and
advance. Rebuild is reset plus catch-up.

## 9. Transport

### 9.1 Envelope

Integration events cross service boundaries as the section 4.2 `envelope`, keyed
by the deterministic `delivery_id`. The payload is an integration event produced
by an anti-corruption transform, not an internal event row. Internal refactors
therefore do not become cross-service migrations.

### 9.2 Publication registry

Each service declares its published set as data:
`(aggregate_type, event_type, event_version) -> (topic, transform_id)`. Unlisted
events are unpublished and framework streams are unpublishable. The registry
MUST be exportable as the service's integration-contract artifact; its hash MAY
be carried in broker headers.

### 9.3 Relay

A reactor named `relay:<service>` proposes `CommandDelivery(encode(envelope))`
with journal-derived delivery ids. Its outbox deliverer publishes with partition
key `aggregate_id` and header `delivery_id`. Broker failures classify transient
(unavailable, timeout) or terminal (unknown topic, serialization). The protocol
is at-least-once overall and deduplicated downstream.

### 9.4 Ingest

Decode the envelope, translate the integration event to a local command (skip
unaddressed events), and call `deliverCommand` with the envelope's delivery id.
Broker offsets are cursors; receipts are truth. Commit the broker offset only
after `Applied` or `AlreadyApplied`. Poison messages park on a dead-letter topic
and MUST NOT block the partition.

### 9.5 Broker requirements

The broker must provide per-key ordered, at-least-once delivery with consumer
offsets. Kafka and Redpanda qualify. Classic RabbitMQ queues do not provide
per-key ordering under competing consumers and MUST NOT carry entity-event
topics. The partition-key rule maps per-stream order onto partition order.

## 10. Conformance and testing

### 10.1 Corpus layout

```text
conformance/
  encoding/*.cbor
  kernels/*.cbor
  identity/*.cbor
```

Encoding vectors are `[1, type_name: text, value: any, bytes: bytes]` and
identity vectors are
`[1, source, aggregate_type, aggregate_id, sequence,
delivery_id]`. No corpus
change is valid without a spec change in the same commit, and no semantic spec
change is valid without its corpus update.

### 10.2 Kernel vectors

```cddl
vector = [1, kernel: tstr, given: given, input: any, expect: expect]
given  = [0] / [1, state: bstr]
expect = [0, error: any] / [1, verdict: any, state': given]
```

The engine suite replays every vector against its pure kernel module. Section
5.5 tables are replayed against reference implementations in the engine test
crate. Comparison is by encoded bytes. Kernel input layouts are documented in
`conformance/README.md`.

### 10.3 Binding tests

- **Codec conformance:** each binding replays encoding and identity vectors
  against its own codecs and asserts byte equality.
- **ABI integration:** each binding runs its behavioral suite against the real
  engine on `":memory:"`; there is deliberately no mock production backend.
- **Error taxonomy:** bindings assert representative section 7.3 class and
  section 4.2 detail encodings.

### 10.4 Build integration

The FFI crate generates and exports the header as a build artifact. The Haskell
binding compiles `capi` imports against it and links the static library. Under
Nix, cbindgen runs in the Rust build and its include and library directories are
fed to Cabal from the same derivation graph. The ABI version is also checked at
runtime by the binding's open wrapper. The conformance corpus is one derivation
consumed by the engine and every binding's check phase.

## 11. Bindings

### 11.1 Requirements

1. **No reimplementation of engine semantics.** Every persisted transition uses
   section 7 operations. Bindings contain no storage backend or kernel decision
   logic except the exhaustive binding-side tables in section 5.5.
2. No transaction spans the ABI.
3. Each binding provides an idiomatic typed `EventSourced`, `Store`,
   `Projection`, `Reactor`, and worker surface with strongly typed ids and the
   section 3.5 `Effect` shape.
4. Verdicts are sealed by visibility; test fabrication lives behind an explicit
   test-only feature or module.
5. Job membership is compile-checked (`Contains` witness or `Member`
   constraint).
6. Protocol values are one-shot where the type system permits: moves in Rust and
   linear values in Haskell.
7. Both bindings implement codec conformance and section 8 exactly.
8. Blocking foreign calls must not hold binding-runtime-critical locks.

### 11.2 Rust binding

The typed layer remains in crate `event-sorcery`: `EventSourced`, `Store`,
`Effect` and `fx`, `DispatchedJob` with sealed verdicts, `jobs!` and `Contains`,
and supervised workers. It lowers onto the same erased facade used by the FFI,
with no private writer path. `cqrs-es` and `apalis` remain implementation
substrate below the facade and worker host respectively; they do not appear in
the cross-binding contract.

### 11.3 Haskell binding

- **FFI:** `CApiFFI` imports the generated header. Every store call is `safe`;
  only `es_buf_free` and `es_abi_version` are `unsafe`. `EsStore` is owned by a
  `ForeignPtr` with close finalization and explicit idempotent close. Blocking
  calls release the capability and require no bound threads.
- **Surface:** retain the typed Haskell layer: `EventSourced` with injective
  type families, `Effect` with `Member job (Jobs entity)`, `StreamKey`, linear
  one-shot commit values, and pure `TestHarness`/`replay`. The backend type
  classes survive as the seam; their sole production instance is the engine
  handle.
- **Removed:** Haskell memory and SQLite backends, every persisted-state
  `decide*` function, the Binary job record codec, the length-prefixed
  stored-job format, and dense checkpoint arithmetic. Their tests target the
  real engine on `":memory:"`.
- **Added:** engine imports and buffer/error marshalling, section 4 codecs, the
  K12-K14 dispatch machine with sealed internal and test-support modules,
  section 8 worker orchestration, and journal-based reactor and projection
  folds.

## 12. Migration deltas (informative)

### 12.1 Engine

1. Carve an erased facade beneath the typed layer, operating on aggregate
   identity and bytes. The typed Rust API lowers onto it.
2. Extract pure kernels from current job and dispatch decisions. Adopt monotonic
   token fencing, remove claim-sequence routing, and store claim count in the
   authoritative record.

## 13. Strictness contract

This system is consumed by financial software where silent corruption is
catastrophic:

- **Events are immutable.** Shapes are permanent; version forward. There are no
  events-table migrations after creation and no updates or deletes except
  compaction.
- **No direct writes to framework tables** outside the engine's
  load-decide-enact paths. Bindings contain no storage code.
- **Numeric integrity.** Kernel arithmetic is checked or saturating as
  specified; overflow surfaces as a typed exhausted error and never wraps.
  Binding folds and projections use checked arithmetic; precision loss is an
  error.
- **Idempotency at every retry boundary:** token fences, receipt checks, verdict
  absorption, monotonic checkpoints, and enqueue payload equality.
- **Terminal states are retained.** The `$job` stream preserves job histories
  for audit.
- **One engine, one store per service, single writer preferred, ABI major
  version matched, and conformance green** are release criteria.
